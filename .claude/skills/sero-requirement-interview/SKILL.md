# SERO 요구사항 인터뷰

Claude Design Export HTML → 컴포넌트 매니페스트 → 순차 인터뷰 → 정합성 검증 → wiki 기록 의 순서로 진행한다.
**절대 요구사항을 지어내지 않는다. 사용자가 답하지 않은 항목은 전부 `TBD`로 남긴다.**

## 0. 사전 준비

1. 대상 HTML 확정: 사용자가 지정한 파일, 없으면 `screens/` 폴더에서 파일 목록을 보여주고 선택받는다.
2. 화면ID 추출: 파일명 또는 HTML 내 표기(예: `EMP-001`)에서 확보한다.
3. 미리보기 안내: 대상 HTML의 **전체 경로를 채팅에 출력**하고, 사용자에게 "이 경로를 클릭하면 미리보기 패널이 열립니다. 화면을 보면서 인터뷰를 진행하겠습니다"라고 안내한다.
4. `wiki/_index.md`와 `wiki/common/공통규칙.md`를 읽어 이 화면에 적용될 수 있는 규칙을 파악해 둔다. 동일 화면ID의 기존 문서(`wiki/screens/<화면ID>*.md`)가 있으면 그것도 읽는다.

## 1. 컴포넌트 매니페스트 생성

HTML을 파싱해 `data-sero-component` 속성이 붙은 모든 요소를 추출한다. (속성이 없으면 `sero-*` class를 fallback으로 사용한다.)

- 파이썬/스크립트 실행이 가능하면 코드로 파싱하고, 불가능하면 HTML을 직접 읽어 추출한다.
- 결과를 `manifest/<화면ID>-manifest.json`으로 저장한다. 형식:

```json
{
  "screenId": "EMP-001",
  "screenName": "사원관리",
  "sourceHtml": "screens/EMP-001-사원관리.html",
  "components": [
    {"order": 1, "component": "SeroSearchForm", "seroId": "search-emp",
     "label": "검색 조건 영역", "children": ["search-dept", "search-name", "search-status", "btn-reset", "btn-search"]}
  ]
}
```

- **순서(order)는 화면 상단 → 하단, 부모 → 자식.** 자식 컴포넌트(SeroInput 등)는 부모(SeroSearchForm 등)의 인터뷰 안에서 함께 다루고 매니페스트에는 children으로 기록한다. 인터뷰 단위가 되는 최상위 컴포넌트는 보통 5~8개다.
- 매니페스트 요약(컴포넌트 목록과 인터뷰 순서)을 사용자에게 보여주고 시작 확인을 받는다.

## 2. 컴포넌트별 순차 인터뷰

매니페스트 순서대로 **한 번에 한 컴포넌트만** 진행한다.

- 각 컴포넌트 시작 시: `[3/7] SeroGrid (grid-employee) — 사원 목록 그리드` 처럼 진행 위치를 표시한다.
- 질문은 `references/sero-components.md`의 해당 컴포넌트 유형 템플릿을 사용한다. 한 턴에 그 컴포넌트의 질문을 묶어서 제시하되, 항목 번호를 붙여 사용자가 골라 답할 수 있게 한다.
- 사용자의 답은 **의미를 바꾸지 않는 선에서 요구사항 문장으로 정리**한다. 해석이 필요한 애매한 답은 재확인한다.
- 답하지 않은 항목은 `TBD`로 기록하고 다음으로 넘어간다. 사용자가 "다음", "스킵"이라고 하면 즉시 다음 컴포넌트로 이동한다.
- 컴포넌트 하나가 끝나면 **정리된 요구사항을 3~6줄로 요약해 보여주고 확인**받은 뒤 다음으로 진행한다.

## 3. 정합성 검증 (매 컴포넌트 확인 단계에서 수행)

컴포넌트 요약을 확정하기 **전에**, 0단계에서 파악한 wiki 규칙 및 기존 화면 문서와 대조한다.

- 충돌 발견 시 다음 형식으로 즉시 제시하고 사용자의 결정을 받는다:

```
⚠ 정합성 충돌
- 신규 요구사항: "그리드 페이지 사이즈는 50건"
- 기존 규칙: [GRID-01] 모든 목록 그리드의 기본 페이지 사이즈는 20건 (wiki/common/공통규칙.md)
어떻게 처리할까요?
  1) 기존 규칙 유지 (신규 요구사항 폐기)
  2) 이 화면만 예외로 등록 (예외 사유 필요)
  3) 공통 규칙 자체를 변경 (영향받는 화면 전체 검토 필요 — 권고하지 않음)
```

- 사용자의 결정과 사유를 요구사항 문서의 `정합성 검토` 섹션에 기록한다. 예외 등록 시 공통규칙 문서에는 손대지 않고, 화면 문서에 예외임을 명시한다.
- 충돌이 아니라도 관련 공통 규칙이 적용되는 항목에는 규칙 ID를 참조로 달아준다. (예: "삭제 동작: 논리 삭제 [DEL-01]")

## 4. 산출물 기록

모든 컴포넌트가 끝나면:

1. `references/sero-components.md` 하단의 **요구사항 문서 템플릿**대로 `wiki/screens/<화면ID>-<화면명>.md`를 작성한다.
2. `wiki/_index.md`의 화면 목록에 한 줄을 추가(또는 갱신)한다.
3. TBD로 남은 항목 목록을 별도로 뽑아 문서 하단 `미결 항목` 섹션에 모으고, 채팅으로도 알려준다.
4. 최종 문서 경로를 채팅에 출력해 사용자가 클릭해 확인할 수 있게 한다.

## 금지 사항

- 사용자가 말하지 않은 요구사항을 추정해서 채우지 않는다. (화면에 보이는 사실을 "현황"으로 기술하는 것은 허용: 예 "그리드 컬럼: 사원번호, 사원명, …")
- 한 턴에 여러 컴포넌트를 한꺼번에 인터뷰하지 않는다.
- 정합성 충돌을 사용자 결정 없이 임의로 해소하지 않는다.
- wiki의 기존 문서를 사용자 확인 없이 수정·삭제하지 않는다. (_index.md 목록 갱신은 예외)
