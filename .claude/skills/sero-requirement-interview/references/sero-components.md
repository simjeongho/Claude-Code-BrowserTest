# SERO 컴포넌트별 인터뷰 질문 템플릿

인터뷰 시 해당 컴포넌트 유형의 질문을 사용한다. 화면에서 이미 확인 가능한 사실(컬럼 구성, 필드 목록 등)은 묻지 말고 "현황"으로 먼저 제시한 뒤, 확인/보완 질문만 한다.

## SeroPageHeader
1. 화면 접근 권한(메뉴 권한 그룹)은?
2. 화면 진입 시 초기 동작(자동 조회 여부, 기본 검색 조건)은?

## SeroSearchForm
1. 각 검색 조건의 기본값은? (예: 부서=로그인 사용자 소속 부서)
2. 필수 검색 조건이 있는가?
3. 조회 버튼 없이 Enter로 조회가 가능해야 하는가?
4. 검색 조건 간 연동이 있는가? (예: 부서 선택 시 하위 조직 포함 여부)
5. 부서/코드성 select의 데이터 소스는? (공통코드, 조직 API 등)

## SeroGrid
1. 데이터 소스(API/도메인)와 조회 조건 매핑은?
2. 페이지당 표시 건수는? (공통 규칙 GRID 계열 확인)
3. 정렬: 기본 정렬 기준은? 정렬은 서버/클라이언트 어느 쪽인가?
4. 행 선택 방식은? (단일/다중, 체크박스 여부)
5. 더블클릭/행 클릭 시 동작은? (상세 이동, 선택만 등)
6. 컬럼별 특이사항: 마스킹, 포맷(날짜/금액), 링크 처리 대상은?
7. 데이터 0건일 때 표시 문구/동작은?

## SeroPagination
1. 페이징 방식은? (번호형/더보기/무한스크롤)
2. 페이지 이동 시 선택 상태 유지 여부는?

## SeroButton / SeroButtonGroup
버튼 하나마다:
1. 클릭 시 동작(호출 API, 이동 화면)은?
2. 노출/활성 조건은? (권한, 행 선택 여부, 상태값)
3. 실행 전 확인(confirm) 문구가 필요한가?
4. 성공/실패 시 피드백(토스트, 재조회 등)은?

## SeroModal
1. 열리는 조건과 닫히는 조건(저장 성공 시 자동 닫힘 여부)은?
2. 필수 입력 항목과 각 항목의 유효성 규칙은? (길이, 형식, 중복 체크)
3. 저장 시 호출 API와 성공 후 동작(그리드 재조회, 포커스 등)은?
4. 작성 중 닫기 시도 시 경고가 필요한가?

## SeroInput / SeroSelect / SeroDatePicker (부모 컴포넌트 인터뷰에 포함)
1. 형식/길이 제한, placeholder 문구는?
2. 코드성 항목의 데이터 소스는?
3. 날짜 항목의 선택 가능 범위 제한은? (예: 미래일 불가)

## SeroToast
1. 표시 시간, 위치, 중복 발생 시 처리 정책은? (보통 공통 규칙으로 위임)

# 요구사항 문서 템플릿 (`wiki/screens/<화면ID>-<화면명>.md`)

```markdown
# <화면ID> <화면명> 요구사항 정의서

- 원본 화면: screens/<파일명>
- 매니페스트: manifest/<화면ID>-manifest.json
- 작성일: YYYY-MM-DD / 작성 방식: Claude 인터뷰
- 상태: 초안 | 검토중 | 확정

## 1. 화면 개요
(한 단락: 화면의 목적, 주요 사용자, 진입 경로)

## 2. 컴포넌트별 요구사항

### 2.x <컴포넌트명> (<sero-id>)
- 현황: (화면에서 확인되는 구성 사실)
- REQ-<화면ID>-NNN: (요구사항 문장) [참조 규칙 ID]
- ...

## 3. 정합성 검토
| 항목 | 신규 요구사항 | 관련 규칙 | 결정 | 사유 |
|---|---|---|---|---|

## 4. 미결 항목 (TBD)
- [ ] REQ-...-NNN: (질문 내용)

## 5. 변경 이력
| 일자 | 내용 | 결정자 |
|---|---|---|
```

요구사항 ID는 `REQ-<화면ID>-001`부터 컴포넌트 순서대로 부여한다.
